## 1. 목적 & 배경

02가 비전 입력 근거데이터(`vision_manifest.parquet` 1,737행 + 폴더구조 유지 썸네일)를 확보하며 "Ollama 캡션·임베딩 실행"을 다음 세션으로 명시 분리했다(02 spec §8). 이 세션이 그 후속이다.

**핵심 검증 대상은 D19/D20 가설**: "영어 캡션 1개 + multilingual 임베딩"으로 한국어 자연어 질의("결혼식 사진", "케이크 먹은 사진")가 검색되는가. 캡션은 곧 *검색 가능한 텍스트 표면*이므로, **프롬프트가 캡션에 무엇을 담게 강제하는지가 한국어 recall을 좌우**한다. 따라서 이 세션의 1차 산출은 **캡션 프롬프트 도출**이다.

EDA는 구현 게이트가 아니라 insight 확보다(01·02 계승). D19가 "관찰"로 나와도 빌드를 막지 않으며, 프롬프트·모델 선택의 근거 데이터를 남긴다.

**이번 세션 스코프**: ① 캡션 입력 이미지군 선행 파악(저비용 픽셀) → ② 프롬프트 3종 × 모델 2개 정성 비교 → ③ 살아남은 조합의 한국어 질의 검색 recall(정량) + D19 판정. **전부 로컬 Ollama, 외부 전송 0**(ADR-0001).

**모델**(전부 설치·로컬):

- 임베딩: `qwen3-embedding:8b`(4.7GB) — model-decisions 권고와 설치가 일치
- 권고 캡션 모델 Qwen3-VL 8B(model-decisions, 미설치)와의 A/B는 골든셋 있는 **빌드 ⑤단계로 위임**. 이 세션은 설치 모델로 프롬프트·모델크기 trade-off를 본다.

## 3. 분석 항목

### §0. 셋업
manifest·썸네일 로드, Ollama 클라이언트(ollama 0.6.2), `seed=42`. 모든 I/O는 try/except flag-skip(01·02 패턴 계승).

### §1. 선행 픽셀 파악 (Ollama 전, 저비용)
"어떤 사진들이고 캡션이 어려울 군은 어디인가"를 파악해 §2 샘플 층화를 정당화한다.
- **manifest 재집계**: `folder_top`·bucket·포맷(jpg/png)·해상도·`match_confidence`
- **썸네일 픽셀 지표**: 밝기(평균 휘도)·노출(히스토그램 클리핑률)·흐림(Laplacian variance)·종횡비
- **산출**: 편집본(png) vs 원본(jpg) 화질차, 저조도·흐림군 식별 → §2 "어려운 축" 후보

### §2. 정성 캡션 비교 (프롬프트 3종 × 모델 2)
- **층화 샘플 24장**: `folder_top`·bucket·포맷·밝기 다양성 + §1이 식별한 *어려운 축*(어두운 png 편집본, 텍스트 많은 사진)을 의도적으로 포함
- 24 × 3프롬프트 × 2모델 = **144 캡션**(26b 72회, e2b 72회)
- 썸네일 + 6캡션을 나란히 표로 출력
- **정성 기준**: 장소·객체·이벤트 포착 / 환각 유무 / 한국어 질의어휘를 영어로 담는가
- **산출**: 프롬프트·모델 강약점, **후보 1~2 조합 선정**

### §3. 정량 검색 recall (살아남은 조합) + D19 판정
- 한국어 질의셋 **12~15개**: 폴더명→한국어 자동초안 + **사용자 보강**. `folder_top`을 정답 weak label
- 평가 풀 **~400장**: 질의 정답 폴더 전부 + 디스트랙터. best 프롬프트로 캡션→임베딩(`qwen3-embedding:8b`)
- 한국어 질의 임베딩 → top-k 검색 → **recall@5 / @10 · MRR**, 프롬프트 간 비교
- **D19 판정**: 한국어↔영어캡션 매칭 작동 여부(PASS / 관찰). image-embedding leg는 비범위

## 4. 비용 전략

- **정성→정량 2단계**가 무차별 전수 캡션을 차단한다: 느린 26b는 24장(72회)에만, 비용 큰 정량 ~400장은 살아남은 조합·주로 빠른 e2b로.
- **캐시(parquet)**: 캡션·임베딩 결과 저장 → 재실행 저렴. 중단 시 캐시 기준 재개.
- **예상**: 정성 26b 72회 + e2b 72회 / 정량 e2b ~400회 + 임베딩 ~400 + 질의 12~15. 노트북에 실측 wall-clock 보고.

## 9. 프롬프트 3종 명세 (전부 영어 출력)

| 변형 | 의도 | 형태 |
|---|---|---|
| **P1 간결** | 1문장 핵심 | 사진을 한 문장으로 묘사 |
| **P2 구조화** | 검색 슬롯 강제 | 장소유형 · 주요객체 · 활동/이벤트 · 보이는 텍스트 · 분위기를 명시 |
| **P3 하이브리드** | 서술+키워드 | 1~2문장 서술 + 검색용 명사 키워드 목록 |

세 변형 모두 동일 24장에 적용, 동일 정성 기준으로 비교. 구체 프롬프트 문구는 구현 시 노트북 상수로 확정.

## 10. 평가 방법 상세

- **정성(§2)**: 사람이 표를 보고 프롬프트·모델별로 [장소/객체/이벤트 포착 · 환각 · 검색어휘 포함]을 체크 → 후보 1~2 조합 선정.
- **정량(§3)**: weak label = `folder_top`. recall@k = 질의 top-k에 정답폴더 사진이 든 비율. MRR로 순위질 보강. **프롬프트 간 상대 비교가 1차 목적**(절대값은 weak label 한계로 참고).

## 11. 가정 & 미해결

- 폴더명이 질의 정답 weak label로 충분하다고 가정(예 `wedding`→"결혼식"). 다의·노이즈 폴더는 질의셋에서 제외/보정.
- 1024px / q90 썸네일이 캡션 입력으로 충분하다고 가정(02 핸드오프 전제). 부족 신호 시 원본 재디코드 옵션.
- recall 절대값은 weak label·풀 구성에 민감 → **프롬프트 간 상대 순위**를 결론으로 삼고 절대 임계는 두지 않는다.
- 정량 단계 모델은 정성 best를 따른다. 모델까지 정량 비교하려면 비용 재논의.
- EXIF 날짜 결손(46.4% 보유)은 캡션 자체와 무관. 날짜 복원(overlap→iCloud `taken_at`, png→파일명/`folder_date_hint`)은 본 인덱싱(빌드 ③) 또는 별도 후속에서 다룬다.
